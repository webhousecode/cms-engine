# F58 — Interactive Islands

> Preact micro-apps that hydrate inline in static pages — tiny, self-contained interactive widgets managed as Interactives (F39) and rendered natively on the site without iframes.

## Problem

Today, Interactives (F39) are standalone HTML files rendered in iframes or as full pages. This works for large visualizations (charts, infographics) but is wrong for small interactive widgets that should feel native to the page:

- **Shopping cart button** — needs access to page DOM and shared state
- **Contact form** — should match page styling, no iframe border
- **Pricing calculator** — inline in the content flow
- **Gated content gate** — wraps existing page content
- **Search widget** — needs to overlay the page

Iframes create barriers: no shared CSS, no DOM access, fixed dimensions, no SEO, scroll issues. The alternative — hardcoding React components in the site source — removes CMS manageability.

The CMS-ENGINE.md §6.4 vision: "Developer marks a block as `interactive: true`. The block's client-side JS is bundled separately. HTML output includes a lightweight loader (~2KB). The island hydrates independently — rest of page is static. Supports Preact, Svelte, or vanilla JS."

## Solution

A build-time and runtime system that:

1. **Authors** Preact islands in the Interactives Manager (F39) — same upload/AI/edit workflow
2. **Bundles** each island into a tiny self-contained JS file (~2-5KB gzipped) via esbuild
3. **Renders** a `<cms-island>` custom element placeholder in static HTML
4. **Hydrates** islands client-side with a lightweight loader script (~1.5KB)

Islands are a **rendering mode** for Interactives — not a separate system. An Interactive can be rendered as iframe (current) OR as island (new). The CMS manages it; the site renders it.

## Technical Design

### 1. Island vs Iframe — When to Use What

| | Iframe (F39 current) | Island (F58 new) |
|---|---|---|
| **Size** | Full pages (10-200KB) | Micro-apps (2-10KB) |
| **Styling** | Self-contained CSS | Inherits page CSS + scoped styles |
| **DOM access** | Isolated | Full page access |
| **Use cases** | Charts, infographics, demos | Buttons, forms, gates, search |
| **SEO** | Hidden from crawlers | Inline, crawlable placeholder |
| **Data** | Self-contained or fetch | Props from CMS data attributes |
| **Framework** | Any (vanilla, Chart.js) | Preact (3KB runtime) |

### 2. Island Anatomy

An island is a Preact component exported as default from a `.tsx` or `.jsx` file:

```tsx
// counter-island.tsx
import { useState } from "preact/hooks";

interface Props {
  initialCount: number;
  label: string;
}

export default function CounterIsland({ initialCount, label }: Props) {
  const [count, setCount] = useState(initialCount);
  return (
    <button onClick={() => setCount(c => c + 1)}>
      {label}: {count}
    </button>
  );
}
```

### 3. Build Pipeline

Each island is bundled independently via esbuild:

```
Source (.tsx)           Bundle (.js)              Loader
┌──────────────┐      ┌──────────────────┐      ┌────────────────────┐
│ counter.tsx  │─────→│ counter.island.js │      │ cms-islands.js     │
│ (Preact JSX) │      │ (~2KB gzipped)   │      │ (~1.5KB)           │
└──────────────┘      │ - Preact inlined │      │ - finds <cms-island>│
                      │ - tree-shaken    │      │ - loads bundle      │
                      │ - self-contained │      │ - hydrates w/ props │
                      └──────────────────┘      └────────────────────┘
```

**esbuild config per island:**
```typescript
import { build } from "esbuild";

await build({
  entryPoints: ["islands/counter.tsx"],
  bundle: true,
  minify: true,
  format: "esm",
  outfile: "public/islands/counter.island.js",
  external: [],                    // inline everything
  alias: { "react": "preact/compat", "react-dom": "preact/compat" },
  jsxFactory: "h",
  jsxFragment: "Fragment",
  define: { "process.env.NODE_ENV": '"production"' },
  treeShaking: true,
});
```

**Expected bundle sizes:**
| Island | Gzipped |
|--------|---------|
| Counter (minimal) | ~1.8KB |
| Cart button + drawer | ~3.5KB |
| Contact form | ~4KB |
| Gated content gate | ~2.5KB |
| Preact runtime overhead | ~3KB (shared, inlined per island) |

### 4. Site-Side Rendering

**HTML placeholder** (generated by site at build/SSR time):

```html
<cms-island
  component="counter"
  props='{"initialCount":0,"label":"Clicks"}'
  src="/islands/counter.island.js"
>
  <!-- Optional: SSR fallback / SEO content -->
  <button disabled>Clicks: 0</button>
</cms-island>
```

**Loader script** (`cms-islands.js`, included once in `<head>`):

```typescript
// ~1.5KB gzipped
const observer = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (!entry.isIntersecting) continue;
    const el = entry.target as HTMLElement;
    const src = el.getAttribute("src");
    if (!src) continue;
    observer.unobserve(el);
    import(src).then((mod) => {
      const { h, render } = await import("preact");  // or inlined
      const props = JSON.parse(el.getAttribute("props") || "{}");
      render(h(mod.default, props), el);
    });
  }
});

document.querySelectorAll("cms-island").forEach((el) => {
  const eager = el.hasAttribute("eager");
  if (eager) {
    // Hydrate immediately
    const src = el.getAttribute("src")!;
    import(src).then((mod) => {
      const { h, render } = await import("preact");
      const props = JSON.parse(el.getAttribute("props") || "{}");
      render(h(mod.default, props), el);
    });
  } else {
    observer.observe(el);
  }
});
```

**Key features:**
- **Lazy by default** — islands hydrate when scrolled into view (IntersectionObserver)
- **`eager` attribute** — for above-the-fold islands (cart button, nav search)
- **SSR fallback** — static HTML inside `<cms-island>` shows before hydration
- **Props via data** — CMS data passed as JSON in `props` attribute

### 5. Integration with F39 Interactives Manager

Islands are managed in the existing Interactives Manager with a new `type` field:

```typescript
interface InteractiveMeta {
  id: string;
  name: string;
  filename: string;          // source: counter.tsx
  type: "html" | "island";   // NEW — determines rendering mode
  bundlePath?: string;        // built: /islands/counter.island.js
  size: number;
  status: "draft" | "published" | "trashed";
  // ...
}
```

**Admin workflow:**
1. Upload `.tsx` file (or create with AI) → type auto-detected as "island"
2. Preview renders in admin via iframe (with Preact dev build)
3. Publish → esbuild bundles to `.island.js`
4. Embed in content via blocks field (`type: "interactive"`, `renderMode: "island"`)

### 6. CMS Data Flow

Islands receive CMS data as props, matching the F39 data separation principle:

```
CMS Collection (e.g. "products")
  ↓ getDocument() at build/SSR time
Next.js Page (server component)
  ↓ serializes data to props attribute
<cms-island props='{"price":299,"name":"Widget"}' ...>
  ↓ loader parses JSON, passes to Preact component
Island Component (client-side Preact)
  ↓ renders interactive UI
Browser
```

### 7. Site Boilerplate Requirements

Sites need these additions to support islands:

```
site/
├── islands/                        # Island source files
│   ├── counter.tsx
│   └── cart-button.tsx
├── public/islands/                  # Built bundles (gitignored)
│   ├── counter.island.js
│   └── cart-button.island.js
├── scripts/build-islands.ts         # esbuild script
└── src/
    └── components/
        └── CmsIsland.tsx            # Helper component
```

**CmsIsland helper component** (for Next.js sites):

```tsx
// src/components/CmsIsland.tsx
import Script from "next/script";

interface CmsIslandProps {
  component: string;
  props?: Record<string, unknown>;
  eager?: boolean;
  fallback?: React.ReactNode;
}

export function CmsIsland({ component, props = {}, eager, fallback }: CmsIslandProps) {
  return (
    <>
      <Script src="/islands/cms-islands.js" strategy="afterInteractive" />
      {/* @ts-expect-error custom element */}
      <cms-island
        component={component}
        src={`/islands/${component}.island.js`}
        props={JSON.stringify(props)}
        {...(eager ? { eager: "" } : {})}
      >
        {fallback}
      </cms-island>
    </>
  );
}
```

**Usage in a page:**

```tsx
import { getDocument } from "@/lib/content";
import { CmsIsland } from "@/components/CmsIsland";

export default function ProductPage() {
  const product = getDocument("products", "widget-pro");
  return (
    <article>
      <h1>{product.data.name}</h1>
      <p>{product.data.description}</p>
      <CmsIsland
        component="cart-button"
        props={{ productId: product.slug, price: product.data.price }}
        eager
        fallback={<button disabled>Add to cart — ${product.data.price}</button>}
      />
    </article>
  );
}
```

### 8. Relation to Shop Plugin (CMS-PLUGIN-SHOP.md)

The shop plugin is the primary consumer of islands:

| Island | Purpose | Size |
|--------|---------|------|
| `CartIsland` | Add to cart, cart drawer, quantity | ~3.5KB |
| `CheckoutBtn` | Stripe checkout redirect | ~2KB |
| `VariantPicker` | Size/color/option selector | ~2.5KB |
| `GateIsland` | Paywall / gated content | ~2.5KB |

F58 provides the infrastructure; the shop plugin provides the islands. The plugin registers its islands via F46 Plugin System hooks, or simply ships pre-built `.island.js` files.

### 9. Performance Budget

| Metric | Target |
|--------|--------|
| Loader script | < 2KB gzipped |
| Preact runtime (per island) | ~3KB gzipped (inlined) |
| Typical island bundle | 2-5KB gzipped |
| Time to interactive (lazy) | 0ms (loads on scroll) |
| Time to interactive (eager) | < 100ms |
| Total JS for page with 3 islands | < 15KB gzipped |

Compare: React runtime alone is ~45KB. A single iframe embed adds ~50-100KB per instance.
